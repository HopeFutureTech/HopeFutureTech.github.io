🔬 OpenClaw 源码深度解析

架构设计与核心技术实现

导读：基于官方文档与源码分析的技术拆解，带你理解"小龙虾"的内部运作机制。

一、整体架构概览

OpenClaw 采用分层架构设计，核心组件包括：

┌─────────────────────────────────────────────────────────────┐ │ 消息渠道层 (Channels) │ │ WhatsApp | Telegram | Slack | Discord | 飞书 | iMessage... │ └──────────────────────┬──────────────────────────────────────┘ │ ┌──────────────────────▼──────────────────────────────────────┐ │ Gateway 控制平面 │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ Session │ │ Agent │ │ Tools │ │ │ │ Manager │ │ Runtime │ │ Registry │ │ │ └──────────────┘ └──────────────┘ └──────────────┘ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ Cron │ │ WebSocket │ │ Control │ │ │ │ Scheduler │ │ RPC │ │ UI │ │ │ └──────────────┘ └──────────────┘ └──────────────┘ │ └──────────────────────┬──────────────────────────────────────┘ │ ┌──────────────────────▼──────────────────────────────────────┐ │ Agent 运行时 (pi-mono) │ │ - 基于 pi-mono 代码库改造 │ │ - 独立 Session 管理 │ │ - 工具流式调用 (Tool Streaming) │ └─────────────────────────────────────────────────────────────┘
核心设计哲学：
• Gateway 是控制平面，Agent 是产品
• Local-first：优先本地运行，数据不上云
• Multi-channel：统一接入 20+ 消息平台

二、Agent Runtime 深度剖析

2.1 基于 pi-mono 的改造

OpenClaw 的 Agent Runtime 源自 pi-mono 代码库，但做了大量改造：

特性 pi-mono OpenClaw
pi-coding agent ✅ 有 ❌ 移除
Session 管理 pi/tau 目录 OpenClaw 自有
工具调用 基础工具 流式工具调用
配置 ~/.pi/agent openclaw.json
// OpenClaw 重写了一套 Session 管理 // Session 文件存储路径： ~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl

2.2 Bootstrap 文件注入机制

每个 Session 的第一轮对话，OpenClaw 会自动注入以下文件内容：

AGENTS.md

操作指令 + 记忆

SOUL.md

人设、边界、语气

TOOLS.md

工具使用说明

IDENTITY.md

名字/头像/emoji

三、Gateway 控制平面架构

3.1 运行时模型

单一常驻进程 ├── WebSocket 控制/RPC ├── HTTP APIs (OpenAI-compatible) ├── Control UI └── 渠道连接管理

3.2 WebSocket 协议

连接握手流程：

Client Gateway │ │ │ ── connect ──> │ │ │ │ <- hello-ok ─- │ (返回 presence/health/stateVersion/uptime) │ │ │ ── req() ────> │ │ <── res() ──── │

3.3 热重载机制

模式 行为
off 不重载
hot 仅热安全变更
restart 需要时重启
hybrid 安全时热重载，需要时重启（默认）

四、Streaming 流式机制（核心创新）

OpenClaw 实现了双层流式架构：

4.1 Block Streaming（块级流式）

Model 输出 ↓ text_delta/events (模型原生流) ↓ EmbeddedBlockChunker (分块器) ├── 低水位: minChars (默认 800) ├── 高水位: maxChars (默认 1200) └── 断点偏好: paragraph > newline > sentence > whitespace ↓ Channel 发送 (Block 回复)

4.2 Preview Streaming（预览流式）

模式 Telegram Discord Slack
off ✅ ✅ ✅
partial ✅ ✅ ✅
block ✅ ✅ ✅
progress ❌ ❌ ✅

4.3 人性化延迟 (Human-like Pacing)

效果：多块回复之间加入 800-2500ms 随机延迟，更像真人打字。

五、Tools 工具系统架构

5.1 三层权限模型

Tool Profile (基础白名单) ↓ Provider-specific Policy (按模型限制) ↓ global allow/deny (最终过滤)

5.2 工具组 (Tool Groups)

group:runtime

exec, bash, process

group:fs

read, write, edit, apply_patch

group:sessions

sessions_list, spawn, status...

group:memory

memory_search, memory_get

5.3 exec 工具安全机制

• elevated 需双重允许
• 仅 sandbox 模式下生效
• 等价于 host=gateway + security=full

六、Session 与 Memory 系统

6.1 Session 存储格式

~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl

格式：JSON Lines，每行一个事件。

6.2 Memory 系统（RAG 实现）

用户查询 ↓ Embedding 编码 ↓ 向量检索 (语义匹配) ↓ 相关记忆片段 ↓ 注入 Prompt 上下文
Memory.md 三层结构：
• 核心事实（长期保留）
• 动态记录（定期清理）
• 归档（可丢弃）

七、多 Agent 与 Subagent 架构

类型 用途
Main Session 主对话线程
Subagent Session 独立任务执行
Cron Session 定时任务

Subagent 通信模型

Main Agent (协调者) ├── spawn Subagent A (任务 1) ├── spawn Subagent B (任务 2) └── spawn Subagent C (任务 3) ↓ 并行执行，结果汇总

八、Cron 与 Heartbeat 调度系统

特性 Heartbeat Cron
触发方式 Session 轮询 定时触发
状态依赖 依赖当前 Session 独立执行
容错性 低 (Session 挂则停止) 高 (独立进程)
